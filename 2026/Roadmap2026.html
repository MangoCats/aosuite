Roadmap 2026

This roadmap replaces the original Roadmap with a concrete development plan targeting today's technology. It preserves the original phasing of TⒶ¹ through TⒶ⁴ and the same application naming (AOG, AOR, AOS, AOE, AOI, AOV), but specifies exact crates, frameworks, hosting targets, and deliverables for each phase. The phases are designed so that each one produces a working, demonstrable system before the next begins.

Repository Structure

A Cargo workspace with the following crate layout:

ao-types — Core data types: DataItem, DataContainer, VBC, shares/coins arithmetic, timestamps, hash wrappers, key types. No I/O, no async. This is the no_std-compatible foundation that everything else depends on.

ao-crypto — Cryptographic operations: key generation, signing, verification, hashing, separable item handling. Wraps ed25519-dalek, sha2, blake3. Also no_std-compatible.

ao-chain — Blockchain logic: genesis block creation, block construction, assignment agreement validation, UTXO tracking, recording fee calculation, share expiration. Depends on ao-types and ao-crypto. Uses rusqlite for persistence.

ao-recorder — The AOR server binary. Axum HTTP + SSE + MQTT. Hosts one or more blockchain issues, accepts assignment agreements, constructs and publishes blocks, serves query requests, handles pico-payment query accounts.

ao-validator — The AOV binary. Periodically fetches recent blocks from configured AOR instances, hashes them together, publishes anchor proofs.

ao-cli — Command-line tools: genesis block creator (AOG), key management, chain inspection, test transaction generation.

ao-web — React PWA serving AOE, AOS, and AOI views. Communicates with ao-recorder over HTTPS and WebSocket. Manages private keys in browser.

Phase 1: Foundation (Weeks 1–6)

Build the ao-types and ao-crypto crates, plus the genesis block creator CLI tool. At the end of this phase, you can create a genesis block, generate key pairs, sign data, and serialize/deserialize all on-chain data structures. This corresponds to the data layer that the original C++ codebase partially implemented, but done cleanly in Rust with comprehensive tests.

Deliverables

ao-types crate: Variable byte coding (VBC) encoder/decoder matching the original specification in VariableByteCoding.html. DataItem and DataContainer types with serde support for both the binary on-chain format and JSON. Share and coin arithmetic using num-bigint and num-rational, implementing the fixed-point fraction scheme from SharesCoins.html including recording fee calculation per the formula in Fees.html. Timestamp type with signing-time semantics per R8. Item type code registry loaded from a Rust version of byteCodeDefinitions.json, with the separability bit convention from SeparableItems.html.

ao-crypto crate: Ed25519 key pair generation, signing, and verification via ed25519-dalek. SHA2-256 and BLAKE3 hash computation. Signature type that bundles the signature bytes, signing timestamp, and reference to what was signed per R1. Separable item hashing: replace separable DataItems with their hashes before computing a container's signature, per R1b3. Key-never-reuse enforcement utilities per KeyReuse.html.

ao-cli (partial): ao genesis command that creates a genesis block with configurable parameters per Parameters.html: starting shares, number of coins, recording fee rate, share expiration interval, supported signature types. Outputs the genesis block in both binary and JSON. ao keygen command that generates an Ed25519 key pair and prints the public key. ao inspect command that reads a binary block and pretty-prints its contents.

Test suite: Property-based tests (proptest) for VBC round-trip encoding of all i64 values. Round-trip serialization tests for every DataItem type. Signature verification tests including deliberately corrupted signatures. Share arithmetic tests verifying that recording fees produce correct coin inflation. Cross-validation against any test vectors recoverable from the original C++ test suite.

Acceptance criteria

cargo test passes on x86_64 and aarch64 (cross-compiled). A genesis block can be created, serialized to binary, deserialized, and re-serialized to produce identical bytes. A key pair can sign a DataContainer and the signature can be verified. Recording fee arithmetic matches hand-calculated examples from Fees.html.

Phase 2: Single-Chain Recorder — TⒶ¹ (Weeks 7–14)

Build the ao-chain crate and the ao-recorder server. At the end of this phase, a single AOR server can host a blockchain issue, accept signed assignment agreements, validate them, record them in new blocks, and serve the chain to clients. This is the TⒶ¹ milestone from the original roadmap: single recorder, offer-and-acceptance signed share assignments, recording tax, all checking done by the single recorder.

Deliverables

ao-chain crate: Genesis block loading and validation. UTXO database: a SQLite table mapping sequence IDs to public keys, share amounts, and spent/unspent status per the atomic assignment model from AtomicAssignment.html. Block construction: given a set of validated assignment agreements, assemble a new block with correct sequence numbering (CB_FIRST_ID_SEQ_NUM), page layout, recording fee deduction, hash chaining to previous block, and blockmaker signature. Assignment agreement validation: check all signatures per R4, verify share availability against UTXO, verify timestamps per R8, verify recording bid sufficiency, verify no key reuse, verify recording deadline not passed. Share expiration sweep: when constructing a new block, expire any shares whose age exceeds the genesis-configured limit, per Expiration.html.

ao-recorder binary: Axum HTTP server with the following endpoints:
GET /chain/{id}/info — Chain metadata: genesis hash, current block height, current shares outstanding, recording fee rate, estimated recording lag.
GET /chain/{id}/blocks?from={seq} — Download blocks from sequence number onward. Pico-payment query account required per the throttling scheme from IslandLife.html.
GET /chain/{id}/utxo/{seq_id} — Check whether a specific sequence ID's shares are available for assignment.
POST /chain/{id}/submit — Submit a signed assignment agreement for recording. Returns acceptance or rejection with reason.
GET /chain/{id}/events — SSE stream of new block notifications.
Configuration via TOML file specifying which chain(s) to host, data directory, listen address, optional TLS. Designed to host multiple independent chain issues on a single server instance (Gene hosting both BCG and RFM).

ao-cli (extended): ao assign command that constructs an assignment agreement: specify giver sequence ID, receiver public key, amount, recording bid. Signs with the giver's private key. Outputs the partial agreement for the receiver to countersign. ao accept command that countersigns an assignment agreement with the receiver's private key and submits it to an AOR server. ao balance command that queries an AOR server for all unspent shares held by a given public key.

Integration tests: Spin up an AOR server in-process, create a genesis block with 10 shares, assign shares between generated key pairs through the HTTP API, verify UTXO updates, verify recording fees produce correct coin inflation, verify that double-spend attempts are rejected, verify that expired shares are swept.

Hosting target

The recorder cross-compiles to aarch64-unknown-linux-gnu via cross-rs and runs on a Raspberry Pi 4 or 5. A GitHub Actions workflow builds both x86_64 and aarch64 release binaries on every tagged commit. A minimal Dockerfile and a systemd unit file are provided for deployment on either a Pi or a cloud VPS.
*At this point, deployment is manual: copy the binary, create a data directory, write a config TOML, start the service. No orchestration needed for a single-recorder setup.


Acceptance criteria

Two CLI users on different machines can transfer shares through a single AOR server over HTTP. The AOR server runs on a Raspberry Pi for 72 hours under continuous test load (one assignment per second) without memory growth or data corruption. A chain with 100,000 recorded assignments can be fully validated from genesis in under 10 seconds on a Pi 4.

Phase 3: Vendor and Consumer Apps — AOS + AOE (Weeks 15–22)

Build the ao-web PWA with vendor (AOS) and consumer (AOE) views. At the end of this phase, Bob can set up his grill cart with a tablet running AOS, and Alice can walk up with her phone running AOE and pay for a plate of curry goat by assigning a BCG share back to Bob. This is where Island Life comes to life as a working demo.

Deliverables

Key management in browser: Ed25519 key pair generation using Web Crypto API (or tweetnacl-js as fallback). Private keys stored in IndexedDB, encrypted with a user-chosen passphrase via PBKDF2. Key backup as an encrypted file download. Key import from file. Display of public key as a QR code for in-person exchange.

AOE (consumer) view: Dashboard showing balances across all chain issues the user holds shares in, displayed as coin values with cash-equivalent estimates. Chain discovery: enter an AOR server URL or scan a QR code to add a new chain. Assignment flow: select an issue, enter amount, scan or paste recipient public key, review transaction details including recording fee, confirm with passphrase, submit to AOR, display confirmation when recorded. SSE subscription for real-time balance updates when incoming assignments are recorded. GPS-aware vendor map:
*Vendors who configure a location in their AOS profile appear on the map. This is the feature that lets guests find Bob's grill cart. Location data is published as a separable item in the vendor's chain, so it can be updated without a share assignment.
shows nearby vendors who accept shares in issues the user holds, with current availability status.

AOS (vendor) view: Vendor profile: business name, location (GPS or manual), description, operating hours, photo. These are published as separable items on the vendor's chain. Incoming assignment monitor: real-time display of assignments received, with running daily total. Quick-redeem interface: when a customer presents a share, the vendor taps to confirm receipt of goods/services, and the AOS records this as metadata on the vendor's chain.
*In the simplest case, the customer assigns a BCG share back to Bob's public key. Bob's AOS sees the incoming share, plays a confirmation sound, and Bob hands over the curry goat. The metadata recording is optional and provides a transaction log for Bob's bookkeeping.
Share float display: current number of shares outstanding, with color-coded warning when approaching the vendor's configured limit (like Bob's 200-plate cap). Price card: vendor sets ask prices for new shares, displayed to AOE users browsing the vendor's listing.

Shared components: Transaction history with filtering by date, chain, and counterparty. Settings: preferred currency display, notification preferences, AOR server list. Offline mode: queue outgoing assignments when connectivity is lost, submit when reconnected. The PWA service worker caches the app shell and recent chain state for offline viewing.

AOR WebSocket endpoint: Add a /chain/{id}/ws WebSocket endpoint to the recorder for bidirectional client communication, supporting the real-time interaction patterns needed for the multi-step assignment flow (proposing, countersigning, confirming).

Acceptance criteria

Two phones on the same WiFi network, one running AOS and one running AOE, can complete a share assignment in under 3 seconds from tap-to-confirm to confirmation displayed. The PWA installs to home screen on both iOS Safari and Android Chrome. The AOS vendor map shows a test vendor's location within 50 meters of their actual GPS position. The app functions (read-only, showing cached balances) when the phone is in airplane mode.

Phase 4: Market Making — AOI + Exchange (Weeks 23–32)

Build the AOI (investor/market maker) view and the exchange infrastructure that allows shares in different chain issues to be traded. This is where Eddie, Charlie, Ziggy, and Ted enter the picture, and where the system transitions from single-chain coupon redemption to a functioning micro-economy. This phase includes the first steps toward TⒶ² (multiple chain transactions) using the simplified exchange-agent model before tackling full atomic CAA.

Deliverables

AOI view: Portfolio dashboard across multiple chain issues with bid/ask spread display. Automated trading rules engine matching the patterns described in AssignOnwardInvestor.html: configurable bid ladders, ask ladders, float-sensitive pricing (like Ted's discount when Bob exceeds 100 plates outstanding), and position limits (like Charlie's 24-plate cap). Order book display for each issue showing all visible bids and asks from all market participants. Trade execution: when an AOE user's desired purchase matches an AOI's ask, the AOI automatically countersigns the assignment agreement. ROI tracking: annualized return calculation per issue based on recording fee income (coin inflation) and realized trading gains.

Exchange agent infrastructure: A new data structure: the exchange listing, published as a separable item on an exchange agent's own chain. It advertises the agent's current bid and ask prices for shares in other chains, denominated in the agent's own credits (like Charlie's CCC). AOE and AOS apps discover exchange listings through a lightweight directory: each AOR server publishes a JSON index of all issues it hosts, including links to known exchange agents who trade in those issues. This is the DNS/discovery mechanism discussed in the 2026 Review.

Two-party exchange (simplified CAA): Before implementing the full multi-chain atomic protocol, implement a simpler exchange-agent-mediated model. When Alice wants to buy 1 BCG using CCC, her AOE finds Charlie advertising CCC↔BCG exchange. Alice sends 12 CCC to Charlie (a normal single-chain assignment on the CCC chain). Charlie's AOI, upon seeing the incoming CCC, automatically sends 1 BCG to Alice's designated key (a normal single-chain assignment on the BCG chain). The two assignments are not atomically linked at the protocol level; Charlie absorbs the settlement risk, which is acceptable because Charlie controls both sides and trusts himself.
*This is exactly what the Island Life narrative describes: Charlie pre-purchases BCG from Bob and resells them to guests. The atomicity is provided by Charlie's inventory and trustworthiness, not by protocol-level cross-chain escrow. This works for small amounts and known exchange agents, deferring the complexity of full CAA to a later phase.


MQTT inter-recorder messaging: AOR servers connect to a shared Mosquitto broker and publish new block notifications to per-chain MQTT topics. Exchange agent AOI instances subscribe to topics for all chains they trade in, receiving real-time updates that trigger automated trading responses. This replaces the polling that would otherwise be needed for cross-chain awareness.

Referral fee tracking: When an exchange agent facilitates a trade, the fee structure (Charlie's 5% with Ted, Ziggy's 10%) is recorded as metadata in the assignment agreement. AOI displays net-of-fees returns and supports configurable referral agreements between exchange agents.

Hosting additions

A Mosquitto MQTT broker runs alongside the AOR server, either on the same Pi or as a separate lightweight service on a cloud VPS. The broker configuration is minimal: anonymous access with ACLs restricting publish to authenticated AOR servers, subscribe open to any authenticated client. For a single-island deployment, one Mosquitto instance is sufficient. For multi-island (Charlie connecting Caribbean islands), brokers can bridge to each other using Mosquitto's built-in bridge protocol.

Acceptance criteria

An AOI instance configured with Charlie's trading rules from AssignOnwardInvestor.html automatically executes BCG trades without human intervention. A simulated market with 5 AOI instances trading in 3 chain issues reaches price equilibrium within 200 transactions. An AOE user can purchase BCG using CCC through Charlie's exchange in under 10 seconds end-to-end. The MQTT broker handles 100 messages per second (far beyond any realistic island transaction volume) on a Pi 4.

Phase 5: Validation and Trust — AOV (Weeks 33–38)

Build the ao-validator binary and the trust infrastructure. This is Victor's contribution: independent validation that no AOR server has altered its chain history, anchored to external proof.

Deliverables

ao-validator binary: Configurable list of AOR servers to monitor. Periodically (configurable interval, default hourly) fetches all new blocks from each monitored chain since the last check. Computes a rolled-up hash: hash together the most recent block hash from every monitored chain. Stores the rolled-up hash locally with timestamp. Optionally publishes the rolled-up hash to an external anchor.
*The original design suggested publishing to a major public blockchain. In practice, publishing a hash to any tamper-evident public record works: a Bitcoin OP_RETURN transaction, an Ethereum event log, or even a signed post to a public append-only log service. The point is that after publication, any alteration of any monitored chain would be detectable by re-computing the rolled-up hash and comparing to the anchored value.
Alerting: when a monitored AOR server returns chain data that does not match previously fetched blocks (evidence of history alteration), the validator emits an alert via MQTT and optionally webhook/email.

Chain integrity API: The validator exposes an HTTP endpoint that AOE/AOS/AOI clients can query: GET /validate/{chain_id} returns the last validated block height, timestamp, and the anchor publication reference. AOE displays a trust indicator (validated up to block N, anchored at time T) on each chain's detail view.

AOR cross-reference: AOR servers are configured with one or more AOV validator URLs. When serving chain info to clients, the AOR includes the validator's endorsement. Clients can independently verify by querying the validator directly. This is the pattern described in Island Life where AOR hosts configured their servers to refer guests' AOE to Victor's AOV.

Vendor credential linking: Per the Island Life narrative about Bob's food safety certificates, implement a mechanism for vendors to include verifiable credential references as separable items in their chain. These are URLs pointing to external records (community college completion records, professional licenses) along with a hash of the credential content at time of linking. AOE displays these credentials on the vendor's profile, with a visual indicator of whether the linked content still matches its recorded hash.

Acceptance criteria

An AOV instance monitoring 10 chain issues detects a simulated history alteration (a single modified byte in a past block) within one polling interval and emits an alert. The rolled-up hash can be independently verified by a third party using only the anchor publication and the current chain data from the AOR servers. The validator runs continuously for 30 days on a Pi without memory growth.

Phase 6: Atomic Multi-Chain Exchange — TⒶ² (Weeks 39–48)

Implement the full Conditional Assignment Agreement (CAA) protocol from MultichainExchange.html, enabling trustless atomic exchanges across independent chains without requiring an exchange agent to absorb settlement risk. This is the most technically demanding phase and can be deferred indefinitely if the exchange-agent model from Phase 4 proves sufficient for real-world use.

Deliverables

CAA data structures in ao-types: Conditional assignment agreement container extending the basic assignment agreement with: ordered list of concerned chains (by genesis hash), escrow period, per-chain assignment terms, and slots for recording proofs from each chain. State machine for CAA lifecycle: proposed → signed → recording-in-progress → binding → finalized, with timeout transitions to expired at each stage.

Escrow support in ao-chain: When a CAA is recorded on a chain, the shares involved are placed in escrow: marked as unavailable for other assignments but not yet transferred. The escrow has a deadline. If the binding proof (recording on all concerned chains) arrives before the deadline, the assignment takes effect. If the deadline passes without binding proof, the escrow is released and the shares return to their previous owner. This requires a new UTXO state: unspent, escrowed-pending-CAA, or spent.

CAA coordinator in ao-recorder: The AOR server acts as coordinator for CAAs involving its hosted chains. When a fully-signed CAA arrives, the AOR records it (placing shares in escrow), signs the recording block, and forwards the CAA with appended recording proof to the next chain's AOR in the ordered list. When the AOR receives a CAA with recording proofs from all other concerned chains, it records the binding proof, finalizing the escrow. The Ouroboros recording sequence described in MultichainExchange.html is implemented as a series of HTTP POST calls between AOR servers, with MQTT notifications to interested clients at each step.

Timeout and recovery: If any AOR server in the chain is unreachable during CAA recording, the coordinator retries with exponential backoff until the escrow deadline. If the deadline is reached without full recording, the coordinator records an escrow release on all chains where recording did occur. AOE/AOS/AOI clients are notified of CAA failure via SSE/WebSocket, and the assignment is shown as expired. The escrow release mechanism is the critical safety property: no shares can be permanently locked by a failed CAA.

AOE multi-chain transaction flow: When Alice wants to give Bob 1 BCG and she only holds CCC, the AOE presents this as a single action: "Give 1 BCG to Bob (costs 12 CCC via Charlie)." Behind the scenes, the AOE constructs a CAA involving both the CCC and BCG chains, collects signatures from all parties (Alice, Charlie, Bob), and submits it for coordinated recording. Alice sees a progress indicator as the CAA moves through the recording sequence, and a final confirmation when it becomes binding. If the CAA fails, Alice sees a clear explanation and her CCC balance is restored.

Acceptance criteria

A three-party, two-chain CAA (the Alice/Charlie/Bob scenario from MultichainExchange.html) completes successfully end-to-end in under 30 seconds when both AOR servers are responsive. When one AOR server is made unreachable during a CAA, the escrow times out and all shares are correctly released on the responsive chain within the configured deadline. No shares are lost or double-spent under any combination of server failures, network partitions, and client disconnections, verified by a chaos-testing harness that randomly kills and restarts AOR processes during CAA recording.

Phase 7 and Beyond

The following are not scheduled but represent the natural evolution after the core system is functional:

TⒶ³ — Multiple recorders: Extend the Ⓐ¹ single-recorder model to support multiple competing recorders for a single chain. This introduces temporary forks and merges, requiring a fork-choice rule (longest chain, or highest-residual-value chain as described in Architecture.html). Share incentives for recorders replace the flat hosting fee. This is a significant architectural change and should only be undertaken if single-recorder operation proves to be an unacceptable single point of failure for real deployments.

TⒶ⁴ — Underwriters and error checkers: Recognized shareholders may provide underwriting (staking their own shares to guarantee transaction validity) and error checkers may claim underwriting shares when they find invalid transactions. This completes the trust-but-verify model from BlockchainBasics.html and provides the economic security guarantees that make the system trustworthy at larger scale without proof of work.

AOL — Licensing and learning records: On-chain records of educational achievement, professional licensing, complaints and resolutions, as referenced throughout the Island Life narrative (Bob's food safety certificate, the community college courses).

AOA — Artist shares: Genesis blocks representing digital artworks with patron shares, as described in ArtPatronage.html and Applications.html.

Pico-payment query throttling: The full implementation of per-query micro-charges described in IslandLife.html and Fees.html, to protect AOR servers from abusive global traffic when an island issue attracts unwanted attention from speculators like Quincy.

W3C DID integration: Decentralized identifiers for vendor and exchange agent identity, providing machine-verifiable trust assertions that scale beyond the informal personal-knowledge model of the original Island Life community.
