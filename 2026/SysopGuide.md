# AO Recorder Sysop Guide

This guide is for the person running an AO Recorder server — the Mako on Likiep, the Wanjiku in Kiambu, or anyone operating the infrastructure that keeps an Assign Onward chain running. Read it start to finish before your first deployment; refer back to specific sections as needed.

## What You Are Running

The AO Recorder is a lightweight blockchain server. It accepts signed transactions from clients (wallets), validates them, records them in blocks, and stores the chain data in a SQLite database. It serves the chain data back to clients over HTTP. One recorder can host multiple independent chains.

**Resource profile**: A Raspberry Pi 5 with an SSD can comfortably handle dozens of chains with hundreds of transactions per day. A cloud VM handles thousands. The recorder is single-process, event-driven (tokio async), and stores all chain data in SQLite databases — one `.db` file per chain plus an optional blob directory for photo/document attachments.

## Installation

### Raspberry Pi OS (64-bit) / Debian / Ubuntu

```bash
sudo dpkg -i ao-recorder_*.deb
```

The `.deb` package installs:
- `ao-recorder` binary in `/usr/bin/`
- systemd service unit `ao-recorder.service`
- Default config template at `/etc/ao-recorder/recorder.toml`
- Data directory at `/var/lib/ao-recorder/`
- System user `ao-recorder` (runs the service)

### Windows

Run the `.msi` installer. It installs to `Program Files\AssignOnward\` and creates a Windows Service. The data directory is `%ProgramData%\ao-recorder\` and the config file is alongside the binary.

### From Source

```bash
cargo install --path 2026/src/ao-recorder
```

## First-Time Setup

### 1. Generate configuration

```bash
ao-recorder init
```

This creates `recorder.toml` with sensible defaults and a fresh blockmaker keypair. It also creates the `data/` directory. On package-installed systems, this runs automatically on first start if no config exists.

### 2. Edit the configuration

Open `recorder.toml`. The key settings:

```toml
host = "0.0.0.0"       # Listen address ("0.0.0.0" for all interfaces)
port = 3000             # HTTP port
blockmaker_seed = "..." # Ed25519 seed (generated by init)
data_dir = "data"       # Where databases and blobs are stored
```

For a single chain, uncomment and set:
```toml
db_path = "data/mychain.db"
genesis_path = "genesis.bin"
```

For multiple chains, add `[[chains]]` sections — see the generated config for examples.

### 3. Verify the setup

```bash
ao-recorder doctor
```

This runs a checklist of post-install diagnostics:
- Binary executes
- Config file is valid
- Blockmaker seed is a valid Ed25519 key
- Data directory exists and is writable
- Port is available
- SQLite is functional
- Disk space is adequate

Fix any `[FAIL]` items before starting the service.

### 4. Start the service

**Systemd (Linux/Pi):**
```bash
sudo systemctl start ao-recorder
sudo systemctl enable ao-recorder  # start on boot
```

**Windows:**
```powershell
sc start ao-recorder
```

**Manual (any platform):**
```bash
ao-recorder recorder.toml
```

### 5. Verify it's running

Open `http://localhost:3000/health` in a browser. You should see a JSON response with `"status": "ok"`. Open `http://localhost:3000/dashboard` for the visual overview.

## Monitoring

### The Dashboard

Open `http://<your-recorder>:3000/dashboard` in a browser. The dashboard auto-refreshes every 10 seconds and shows:

- **Status indicator**: Green (OK), amber (degraded), or red (error)
- **RAM usage**: System memory with a visual gauge
- **Disk usage**: Data directory filesystem with a visual gauge
- **Chain count**: Number of hosted chains
- **UTXO total**: Sum of all unspent transaction outputs
- **Chain table**: Per-chain details including block height, last block age, UTXO count, and database size

The dashboard requires no authentication by default — it is designed for LAN-only use or deployment behind a reverse proxy with authentication.

### Health API

`GET /health` returns machine-readable JSON:

```json
{
  "status": "ok",
  "uptime_seconds": 86400,
  "version": "0.1.0",
  "chains": [
    {
      "chain_id": "abc123...",
      "symbol": "TST",
      "block_height": 142,
      "last_block_age_seconds": 3600,
      "utxo_count": 85,
      "db_size_bytes": 524288
    }
  ],
  "system": {
    "ram_used_bytes": 1073741824,
    "ram_available_bytes": 3221225472,
    "cpu_load_percent": null,
    "disk_free_bytes": 50000000000,
    "disk_used_bytes": 10000000000
  },
  "capacity": {
    "estimated_assignments_per_second": null,
    "estimated_days_until_disk_full": null
  }
}
```

You can point external monitoring tools (Uptime Kuma, Prometheus, Grafana, or a simple cron + curl script) at this endpoint.

### Logs

The recorder logs structured events via the `tracing` framework. Key log events:

- **Chain loaded / Genesis loaded**: On startup, each chain that initializes successfully
- **Block recorded**: Every new block with chain_id, height, and assignment count
- **Chain created**: When a new chain is created via the API
- **Disk space low/critically low**: When data filesystem drops below warning/error thresholds
- **Chain stale**: When a chain hasn't recorded a block within the configured threshold
- **Memory baseline**: Periodic RSS and system memory snapshot (hourly by default)
- **MQTT/validator events**: Connection status and polling results

On systemd, view logs with:
```bash
journalctl -u ao-recorder -f          # follow live
journalctl -u ao-recorder --since today  # today's logs
```

On Windows, check the Event Log or redirect output to a file.

## Common Problems and Solutions

### Dashboard shows "Unreachable"

The dashboard page cannot reach the `/health` endpoint.

**Causes and fixes:**
1. **Recorder not running**: Check `systemctl status ao-recorder` (Linux) or `sc query ao-recorder` (Windows). Start it if stopped.
2. **Wrong URL**: Ensure you're connecting to the correct host and port.
3. **Firewall**: The recorder port (default 3000) may be blocked. Check `ufw status` (Linux) or Windows Firewall settings.

### Status is "degraded"

**No chains hosted**: The recorder started but no chains are configured or loaded. Check your `recorder.toml` for `db_path`/`genesis_path` or `[[chains]]` entries. Check logs for genesis loading errors.

**Low disk space**: The data filesystem has less than ~1 GB free. Free up space by removing old logs, unused files, or pruning blob storage. See "Disk Management" below.

### Status is "error"

**Critically low disk**: Less than 100 MB free on the data filesystem. The recorder may fail to record new blocks. Immediately free disk space.

### Chain shows old "Last Block Age"

A chain that hasn't recorded a block in over 24 hours is flagged as stale.

**Causes:**
1. **No activity**: If nobody is submitting transactions, this is normal. A cooperative chain during off-season, for example.
2. **Client disconnected**: Clients may have lost connectivity or switched to a different recorder.
3. **Blockmaker error**: Check logs for validation or recording errors on that chain.

### High UTXO count growing over time

Every assignment creates new UTXOs (one per receiver plus change). Old UTXOs are marked spent but remain in the database. Rapid UTXO growth means many transactions — this is normal for active chains. The UTXOs table is indexed and queries remain fast up to millions of rows.

### Memory usage growing over time

The recorder's RSS should stabilize within the first hour. If it grows steadily over days:
1. Note the growth rate from the periodic memory baseline logs
2. Restart the recorder (this is safe — all state is in SQLite)
3. Monitor if the growth resumes
4. Report the issue with your memory baseline logs

### Port already in use

`ao-recorder doctor` reports the port is unavailable.

**Fixes:**
1. Another recorder instance is running — stop it first
2. Another service uses that port — change the `port` in `recorder.toml`
3. On Linux: `lsof -i :3000` to find what's using the port

## Disk Management

### Database growth

Each chain's SQLite database grows with every block. A typical assignment creates a block of ~500 bytes. At 100 transactions per day, a chain grows about 50 KB/day or ~18 MB/year. Ten chains at that rate: ~180 MB/year.

### Blob storage

Photo and document attachments are stored in `data_dir/blobs/` as content-addressed files. A 2 MP photo compressed to WebP is typically 100–300 KB. At 20 photos per day, blob storage grows ~3–6 MB/day or ~1–2 GB/year.

To check blob storage size:
```bash
du -sh /var/lib/ao-recorder/blobs/
```

Blob pruning is not yet implemented. If blob storage becomes too large, blobs can be manually removed from the filesystem — the chain data (hashes) remains intact, but clients will get 404 when trying to view pruned attachments.

### Backup

The critical data is the SQLite database files and the blob directory. Back up the entire `data_dir`:

```bash
# Simple rsync backup
rsync -avz /var/lib/ao-recorder/ backup-server:/backups/ao-recorder/

# Or copy the directory
cp -r /var/lib/ao-recorder/ /mnt/backup/ao-recorder-$(date +%Y%m%d)/
```

SQLite databases are safe to copy while the recorder is running (WAL mode handles concurrent access). For extra safety, you can use `sqlite3 .backup`:

```bash
sqlite3 /var/lib/ao-recorder/chain.db ".backup /tmp/chain-backup.db"
```

## TLS / HTTPS

The recorder serves plain HTTP. For production deployments exposed to the internet, put it behind a reverse proxy with TLS.

**Recommended: Caddy** (automatic Let's Encrypt certificates):

```
ao.example.com {
    reverse_proxy localhost:3000
}
```

Save as `Caddyfile`, run `caddy run`. Caddy handles certificate issuance and renewal automatically.

**Alternative: nginx** with certbot:
```nginx
server {
    listen 443 ssl;
    server_name ao.example.com;
    ssl_certificate /etc/letsencrypt/live/ao.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/ao.example.com/privkey.pem;
    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

For LAN-only deployments (Tia's store, Riuki cooperative), TLS is not required.

## Performance Benchmarking

Run the built-in benchmark to establish a baseline for your hardware:

```bash
ao-recorder bench recorder.toml
```

This measures:
- **Block storage throughput**: SQLite insert speed (the recording bottleneck)
- **Ed25519 signature speed**: Sign + verify operations per second

Typical results:
- Raspberry Pi 5 (SSD): ~2,000–5,000 blocks/sec, ~1,500 sign+verify/sec
- Cloud VM (2 vCPU): ~5,000–15,000 blocks/sec, ~3,000 sign+verify/sec
- Desktop (modern CPU): ~15,000–50,000 blocks/sec, ~5,000 sign+verify/sec

At 100 transactions per day, even a Pi has 100x headroom.

## Configuration Reference

### Main settings

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `host` | string | `"127.0.0.1"` | Listen address |
| `port` | u16 | `3000` | Listen port |
| `blockmaker_seed` | string | *required* | Ed25519 seed (64 hex chars) |
| `data_dir` | string | none | Persistent storage directory |
| `db_path` | string | none | Single-chain SQLite database path |
| `genesis_path` | string | none | Single-chain genesis block file |
| `dashboard` | bool | `false` | Enable `/dashboard` endpoint |

### Chain config `[[chains]]`

| Key | Type | Description |
|-----|------|-------------|
| `db_path` | string | SQLite database path for this chain |
| `genesis_path` | string | Genesis block file for this chain |
| `blockmaker_seed` | string | Optional per-chain blockmaker seed |

### MQTT `[mqtt]`

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `host` | string | *required* | MQTT broker hostname |
| `port` | u16 | `1883` | MQTT broker port |
| `client_id` | string | `"ao-recorder"` | MQTT client ID |
| `topic_prefix` | string | `"ao/chain"` | Topic prefix for block notifications |

### Validators `[[validators]]`

| Key | Type | Description |
|-----|------|-------------|
| `url` | string | Validator API base URL |
| `label` | string | Human-readable label |

### Alerts `[alerts]`

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `disk_warn_percent` | f64 | `10.0` | Disk free % to trigger warning |
| `disk_error_percent` | f64 | `5.0` | Disk free % to trigger error |
| `stale_chain_seconds` | u64 | `86400` | Seconds without a block before stale alert |
| `memory_log_interval_seconds` | u64 | `3600` | Seconds between memory baseline logs |
| `webhook_url` | string | none | HTTP POST URL for alert notifications |

### Webhook payload

Alert webhooks POST JSON:
```json
{
  "event": "disk_warning",
  "message": "Disk space low: 8.2% free (12345678 bytes)",
  "timestamp": 1709827200
}
```

Event types: `disk_warning`, `disk_critical`, `chain_stale`.

## Operational Checklist

### Daily (or verify via dashboard)
- [ ] Dashboard status is green
- [ ] No chains stale (unless expected)
- [ ] Disk usage under 80%

### Weekly
- [ ] Check logs for recurring errors: `journalctl -u ao-recorder --since "1 week ago" -p err`
- [ ] Verify backup is current

### Monthly
- [ ] Run `ao-recorder bench` and compare with baseline
- [ ] Review blob storage growth: `du -sh data/blobs/`
- [ ] Check for ao-recorder updates

### After OS updates or hardware changes
- [ ] Run `ao-recorder doctor`
- [ ] Run `ao-recorder bench` to re-baseline
- [ ] Verify dashboard shows expected chain data
